---
title: "Token Identity"
description: "How Makx uses .makx domains to give every token a verified on-chain identity."
---

Every token launched through Makx gets a `.makx` domain automatically. This gives each token a human-readable name, verified origin, on-chain metadata, and a permanent web address — no backend required.

## Key Guarantees

- **Verified Origin** — Only the Makx factory and approved factories can issue .makx domains. If a token has a .makx domain, it was launched through Makx.
- **Lifetime Domains** — .makx domains never expire and always resolve to the token's contract address.
- **Immutable Core Data** — A token's address, name, and symbol cannot be changed after registration, preventing impersonation.

## How it works

When you launch a token called `PEPE`, the factory registers `pepe.makx` and writes all metadata to the resolver in a single atomic transaction. The result:

| Chain | Domain | URL |
|---|---|---|
| Base | `pepe.makx` | `makx.fun/pepe` |
| BNB Chain | `pepe.makx` | `makx.fun/pepe` |

Since tickers are unique on the launchpad, there can only ever be one `pepe.makx`.

## Architecture

```
                        +-----------------+
                        |  Makx Factory   |
                        | (approved only) |
                        +--------+--------+
                                 |
                                 | register()
                                 v
+---------------+       +------------------+       +---------------------+
|  ENS Registry | <---- | MakxRegistrar    | ----> | MakxTokenResolver   |
| (name tree)   |       | (orchestrator)   |       | (token data store)  |
+---------------+       +------------------+       +---------------------+
                                                            ^
                                                            |
                                                   +--------+--------+
                                                   | MakxUniversal   |
                                                   | Resolver        |
                                                   | (viem/web3      |
                                                   |  compatibility) |
                                                   +-----------------+
```

- **MakxRegistrar** — Orchestrates registration. Only approved factories can call it. Creates ENS records and initializes token data in a single atomic transaction.
- **MakxTokenResolver** — Stores and serves token data. Implements standard ENS resolver interfaces so any ENS-compatible client can query it.
- **MakxUniversalResolver** — Enables standard viem/ethers.js calls like `getEnsAddress()` and `getEnsName()` to work with .makx domains.

## What's stored on-chain

Every token's .makx domain includes these records:

### Immutable records (set at registration)

| Key | Description |
|---|---|
| `name` | Token display name |
| `symbol` | Token ticker symbol |
| `token.symbol` | Token ticker symbol |
| `token.decimals` | Token decimals (e.g. "18") |
| `token.totalSupply` | Total supply (as string) |
| `launchpad.address` | Bonding curve / launchpad address |

The domain's ETH address record (`addr(node)`) points to the **token contract**, not the bonding curve.

### Mutable records (editable by token owner)

| Key | Description |
|---|---|
| `description` | Project description |
| `url` | Website URL |
| `avatar` | Logo/image URL (supports IPFS and NFT references) |
| `com.twitter` | Twitter/X handle |
| `com.github` | GitHub organization |
| `com.discord` | Discord invite |

## Resolving a token

.makx domains are ENS-compatible. You can resolve them using viem with the Makx universal resolver:

```typescript
import { createPublicClient, http, type Chain } from "viem";
import { normalize } from "viem/ens";
import { base } from "viem/chains";

const makxChain: Chain = {
  ...base, // or bsc -- same contract addresses on both
  contracts: {
    ...base.contracts,
    ensRegistry: {
      address: "0xce3830A628674C67e35294128F638dDC9c69d8f8",
    },
    ensUniversalResolver: {
      address: "0x535aeD7b7Cd80Ed9A0835E1Ed29a99d038B8B777",
    },
  },
};

const client = createPublicClient({
  chain: makxChain,
  transport: http(),
});

const tokenAddress = await client.getEnsAddress({
  name: normalize("pepe.makx"),
});

const description = await client.getEnsText({
  name: normalize("pepe.makx"),
  key: "description",
});

const name = await client.getEnsName({
  address: "0x1f9840a85d5aF5bf1D1762F925BDADdC4201F984",
});
```

This works in any ENS-compatible wallet, explorer, or dApp. See the [Frontend Guide](/docs/makx-domains-frontend) for the full integration reference.

## Verifying a token is from Makx

A token has a valid .makx domain if and only if it was registered through an approved Makx factory. Verify with a forward + reverse check:

```typescript
async function isMakxToken(tokenAddress: string): Promise<boolean> {
  const name = await client.getEnsName({ address: tokenAddress });
  if (!name || !name.endsWith(".makx")) return false;

  // Double-check forward resolution matches
  const resolved = await client.getEnsAddress({ name: normalize(name) });
  return resolved?.toLowerCase() === tokenAddress.toLowerCase();
}
```

## Updating metadata

After launch, the token's metadata owner can update mutable text records by calling `setText()` on the `MakxTokenResolver` contract. Metadata ownership can also be transferred to another address. Immutable fields (name, symbol, token address) can never be changed. See the [Contract Integration Guide](/docs/makx-domains-contracts) for details.
